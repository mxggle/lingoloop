# YouTube Media and Transcript Architecture

**Status:** Implemented foundation; production hardening required
**Last reviewed:** 2026-07-06
**Owners:** Player, desktop runtime, media processing
**Applies to:** Tauri desktop application; browser development fallback is explicitly limited

## 1. Purpose

This document is the canonical technical reference for loading YouTube media into
Pawcast. It describes how a YouTube URL becomes:

- controllable playback through the YouTube iframe player;
- a locally cached audio stream used only for analysis and transcription;
- a real multiresolution waveform generated by FFmpeg;
- timed transcript segments sourced from YouTube captions when available;
- an AI-generated transcript when captions are unavailable.

It also records the security model, platform compatibility limits, external-tool
packaging, failure behavior, testing expectations, maintenance procedures, and
known gaps. The distinction between **current behavior** and **target behavior**
is intentional. Do not treat roadmap sections as already implemented.

## 2. Executive summary

YouTube playback and media analysis use separate paths:

1. The YouTube iframe API remains the playback authority. It owns audible
   playback, seeking, duration, volume, playback rate, and player state.
2. The desktop backend runs a pinned `yt-dlp` sidecar to cache the best available
   audio-only stream and JSON3 captions.
3. The existing FFmpeg waveform service analyzes the cached audio and stores
   multiresolution peak levels.
4. The transcript workflow prefers downloaded captions. If no usable caption
   track exists, it sends the cached audio through the user's configured
   transcription provider.

The web build cannot extract YouTube audio. Browser cross-origin boundaries and
the iframe API prevent direct access to media bytes. Web therefore supports
iframe playback only and must not pretend that a synthetic waveform is real.

## 3. Goals and non-goals

### 3.1 Goals

- Produce a waveform derived from the actual audio stream.
- Prefer source captions to avoid unnecessary AI cost and latency.
- Fall back to AI transcription when captions are missing or unusable.
- Keep YouTube playback synchronized with the shared player state.
- Reuse the native waveform pipeline used by local files.
- Cache expensive network and waveform work.
- Keep Tauri and process-spawning details behind the typed `DesktopAPI` boundary.
- Validate every value crossing the frontend/native and network/filesystem
  boundaries.
- Package external tools deterministically with pinned versions and checksums.
- Fail with safe, actionable errors rather than simulated transcript content.

### 3.2 Non-goals

- Replacing the YouTube iframe with a custom media player.
- Supporting YouTube extraction in a normal browser runtime.
- Circumventing DRM, paywalls, account restrictions, region restrictions,
  private-video access controls, or membership controls.
- Automatically importing browser cookies or account credentials.
- Guaranteeing access to every YouTube video or every audio format.
- Treating downloaded captions as word-accurate ground truth.
- Permanently archiving YouTube media.

## 4. Product and policy boundary

The implementation is technically separate from YouTube's official iframe and
Data APIs. The iframe API supports playback control but does not expose audio
bytes. The official caption download API requires OAuth authorization and
permission to edit the video, so it cannot retrieve captions for arbitrary
public videos.

`yt-dlp` provides the missing local extraction capability, but it is not an
official YouTube integration. YouTube's Terms of Service generally restrict
downloading except where the service or rights holder authorizes it. Product and
legal owners must decide where this feature may be distributed and how user
authorization is represented. Engineering must not silently broaden the feature
to authenticated, restricted, or protected content.

Recommended product guardrails before public distribution:

- Make extraction an explicit user action.
- State that users must have permission to process the selected content.
- Do not advertise offline media downloading; the local file is an analysis
  cache supporting the learning workflow.
- Provide cache deletion and retention controls.
- Do not collect or transmit YouTube cookies.
- Re-review YouTube terms and local copyright requirements before each major
  release.

This section records engineering constraints and is not legal advice.

│                                                                        │

│                      ├── prepareYouTubeMedia ── DesktopAPI ── Tauri IPC │

│                                                   youtube_prepare       │

There are two independent external network relationships:

- The iframe communicates with YouTube for user-visible playback.
- `yt-dlp` communicates with YouTube to retrieve audio and captions.

Success in one path does not imply success in the other. An embeddable video may
still fail extraction because of rate limits, client challenges, missing tokens,
or format restrictions.

## 7. Domain contracts

### 7.1 Input

The frontend reduces supported YouTube URLs to a canonical 11-character video
ID. The native command validates the ID again because frontend validation is not
a security boundary.

Accepted native grammar:

```text
^[A-Za-z0-9_-]{11}$
```

The optional caption language is limited to 16 ASCII alphanumeric or hyphen
characters. Invalid language values fall back to `en`.

### 7.2 Desktop capability

Conceptual contract:

```ts
interface YouTubePreparedMedia {
  audioPath: string;
  transcript: Array<{
    text: string;
    startTime: number;
    endTime: number;
  }>;
}

interface DesktopAPI {
  youtubePrepare(
    videoId: string,
    language?: string
  ): Promise<YouTubePreparedMedia>;
}
```

`audioPath` is an internal native cache path. It is not a stable user-facing
identifier and must never be persisted into portable transcript records.

### 7.3 Media identity

- Playback identity: `currentYouTube.id`
- Transcript repository identity: `youtube-<videoId>`
- Native waveform identity: an opaque hash generated from
  `youtube-<videoId>`, e.g. `waveform-fe6338ff`

The native waveform ID is intentionally opaque. Filesystem paths are never used
as cache IDs because they leak local layout and contain path separators rejected
by the native cache validator.

## 8. Audio preparation pipeline

### 8.1 Sequence

```text
UI              Service          Desktop API       Rust             yt-dlp
│                  │                  │               │                 │
│ prepare(id)      │                  │               │                 │
├─────────────────►│                  │               │                 │
│                  │ dedupe id/lang   │               │                 │
│                  ├─────────────────►│               │                 │
│                  │                  ├──────────────►│ validate ID     │
│                  │                  │               │ create cache dir│
│                  │                  │               ├────────────────►│
│                  │                  │               │ bestaudio/best  │
│                  │                  │               │◄────────────────┤
│                  │                  │               │ approve path    │
│                  │                  │◄──────────────┤                 │
│                  │◄─────────────────┤               │                 │
│◄─────────────────┤ cached promise result            │                 │
```

### 8.2 Current command behavior

The native command builds only this URL:

```text
https://www.youtube.com/watch?v=<validated-video-id>
```

It invokes `yt-dlp` using an argument array through `tokio::process::Command`.
No shell is involved. The current audio selector is:

```text
-f bestaudio/best
```

The downloaded container is not forcibly transcoded. Typical results include
Opus in WebM or AAC/M4A. FFmpeg handles both. Avoiding unnecessary transcoding
reduces latency, CPU use, and quality loss.

The output template is scoped to a validated per-video directory:

```text
<active-data-directory>/cache/youtube/<videoId>/<videoId>.%(ext)s
```

After download, the native layer scans the directory, excludes partial-state
files (`.part`, `.ytdl`) and caption files (`.json3`), then approves the selected
audio path in `AppState`. Approval allows the existing waveform command and
`local-media` protocol to read that file without approving arbitrary paths.

### 8.3 Request deduplication

`prepareYouTubeMedia` stores in-flight promises by `<videoId>:<language>`. This
prevents the waveform and transcript consumers from launching duplicate native
downloads in the same frontend process.

Successful promises currently remain in memory for the lifetime of the page.
Failed promises are removed so a later attempt can retry.

Future implementation should move deduplication and locking into Rust as well.
Multiple windows or renderer processes can bypass the frontend map and target the
same cache directory concurrently.

## 9. Caption and transcript pipeline

### 9.1 Caption selection

Audio and caption retrieval are separate operations. Caption failure must not
invalidate a usable audio download.

Current caption arguments request:

```text
--skip-download
--write-subs
--write-auto-subs
--sub-format json3
--sub-langs <requested-language>,en
```

The exact requested language is preferred, followed by English. Wildcard
language matching is intentionally avoided because it can request many automatic
translation tracks and trigger rate limiting.

### 9.2 JSON3 normalization

The native parser reads `events[]` and converts millisecond timing into seconds.
For each event it:

1. reads `tStartMs`;
2. reads `dDurationMs`, defaulting to zero;
3. concatenates `segs[].utf8`;
4. replaces embedded newlines with spaces;
5. drops empty events;
6. emits `{ text, startTime, endTime }`.

JSON3 is an internal YouTube caption representation, not a stable public API.
Parser changes should be expected over time. Malformed or unreadable caption
files currently produce an empty transcript and therefore trigger AI fallback.

### 9.3 Transcript decision tree

```text
Prepared media
    │
    ├── one or more caption segments
    │       └── store as final transcript segments
    │           confidence = 1 (source availability, not linguistic certainty)
    │
    └── no caption segments
            ├── configured provider/key available
            │       └── transcribe cached audio
            │
            └── provider unavailable
                    └── request provider configuration
```

The current `confidence = 1` value means Pawcast accepted a source caption. It
must not be interpreted as evidence that auto-generated captions are perfectly
accurate. A future schema should record provenance explicitly:

```ts
type TranscriptProvenance =
  | { source: "youtube-manual"; language: string }
  | { source: "youtube-auto"; language: string }
  | { source: "ai"; provider: string; model?: string };
```

### 9.4 AI fallback

When captions are absent, the frontend exposes the approved cached file through
the `local-media` protocol, reads it as a `Blob`, and passes it to the existing
transcription service. Long media follows the existing chunked-transcription
threshold and progress behavior.

Current limitation: a requested A-B range is not physically sliced from YouTube
audio before transcription. The prepared blob represents the complete audio
file. Range-aware YouTube transcription must use FFmpeg server-side extraction
or a bounded stream rather than loading the entire file into renderer memory.

## 10. Waveform pipeline

### 10.1 Sequence

```text
WaveformVisualizer
  │
  ├── prepareYouTubeMedia(videoId)
  │       └── cached audioPath
  │
  ├── buildNativeWaveformId("youtube-" + videoId)
  │
  ├── WaveformLoader.getMeta(mediaId)
  │       ├── cache hit → skip analysis
  │       └── cache miss
  │              └── waveformAnalyze(audioPath, mediaId)
  │                     ├── FFprobe metadata
  │                     ├── FFmpeg mono PCM decode
  │                     └── multiresolution cache write
  │
  └── loadForViewport(mediaId, visibleDuration, canvasWidth)
          └── WaveformRenderer.setWaveformData(level)
```

### 10.2 Multiresolution cache

The waveform service decodes audio to mono floating-point PCM, computes the
finest peak level, then generates progressively coarser levels. Metadata and
binary levels are stored under:

```text
<active-data-directory>/cache/waveform/<mediaId>/
├── meta.json
├── level-256.bin
├── level-512.bin
├── ...
└── level-65536.bin
```

`WaveformLoader.loadForViewport` chooses a level based on visible seconds per
pixel and source sample rate. This keeps rendering cost bounded for long media.

### 10.3 Concurrency invariant

Only one native waveform analysis may run per `mediaId` in a renderer process.
All concurrent callers await the same promise.

This invariant is necessary because YouTube reports duration asynchronously.
Duration changes rerun waveform effects. Before shared-promise deduplication, a
second caller failed with “analysis already in progress,” while the first result
was discarded by React effect cleanup. The native cache was valid but the UI
showed a failure. Do not replace the promise map with a boolean or `Set` unless
waiting semantics are preserved.

## 11. Playback synchronization

The downloaded audio is not played. `YouTubePlayer` remains the sole audible
playback source and synchronizes with the shared player store:

- `getDuration()` initializes duration after iframe metadata loads;
- `getCurrentTime()` drives the high-frequency playback clock;
- play/pause state changes update Zustand state;
- external seeks call `seekTo`;
- A-B loop boundaries call `seekTo(loopStart)`;
- master/media volume and mute state map to `setVolume`;
- playback rate maps to `setPlaybackRate`.

Separating playback from analysis avoids trying to reproduce YouTube playback,
ads, authorization, or adaptive streaming. It also introduces a potential timing
difference: the analysis stream and iframe stream may not be byte-identical.
Timing should normally align, but videos with edits, dynamic ad insertion,
alternate audio tracks, or unusual live/post-live behavior require validation.

## 12. Cache architecture

### 12.1 Current cache keys

| Artifact | Key | Persistence |
|---|---|---|
| Downloaded audio | YouTube video ID | Disk |
| Caption JSON3 | Video ID + returned language filename | Disk |
| Native waveform | hash of `youtube-<videoId>` | Disk |
| Prepared-media request | video ID + requested language | Renderer memory |
| Loaded waveform levels | waveform media ID + level | Renderer memory |

### 12.2 Current invalidation

There is no automatic YouTube cache expiration or size limit. A cached video can
be stale if the creator replaces captions, trims the video, changes available
audio tracks, or removes the content.

### 12.3 Target retention policy

Before production release, implement:

- a configurable total cache budget;
- least-recently-used eviction;
- last-access metadata;
- atomic deletion of audio, captions, and related waveform data;
- exclusion of active downloads and analyses from eviction;
- “clear YouTube cache” in settings;
- cleanup of `.part`, `.ytdl`, and abandoned temporary files;
- optional TTL revalidation for captions;
- disk-space preflight before large downloads.

## 13. Trust boundaries and security model

### 13.1 Boundary inventory

| Source | Destination | Risk | Enforcement |
|---|---|---|---|
| User URL | video ID | command/path injection | URL parsing plus native 11-character allowlist |
| Frontend IPC | native command | forged ID/language | native validation before side effects |
| Video ID | YouTube URL | SSRF | fixed HTTPS origin and fixed URL shape |
| Video ID | cache path | traversal | strict character/length allowlist |
| Language | `yt-dlp` argument | argument abuse | length/character allowlist; passed as one array argument |
| Native command | child process | command injection | `Command` with argument array; no shell |
| YouTube response | local filesystem | malformed/large files | scoped cache directory; FFprobe/FFmpeg validation downstream |
| JSON3 | transcript UI | untrusted text/XSS | React text rendering; never inject raw HTML |
| Cached path | local-media protocol | arbitrary file read | explicit canonical path approval |
| Release download | packaged sidecar | supply-chain compromise | pinned version and SHA-256 digest |

### 13.2 Security invariants

- Never pass a raw URL supplied by the user to the native downloader.
- Never use shell command strings.
- Never accept output paths from `yt-dlp` stdout as trusted filesystem roots.
- Never render caption text through `dangerouslySetInnerHTML`.
- Never expose native paths in user-visible errors or telemetry.
- Never auto-import browser cookies.
- Never log cookies, PO tokens, signed media URLs, API keys, or complete command
  output that may contain those values.
- Never weaken path approval to make cached media convenient.
- Never update sidecar URLs without updating verified checksums.

### 13.3 Remaining security work

- Bound child-process runtime with timeouts and cancellation.
- Bound output file size and available disk consumption.
- Capture a limited stderr tail for diagnostics; sanitize before UI/telemetry.
- Use per-video native locks and atomic completion markers.
- Reject symlinks and unexpected file types during cache scanning.
- Validate that the chosen audio path is a regular file inside the canonical
  per-video directory.
- Consider sandboxing or reduced privileges for external tools.
- Add a dependency/license inventory for every packaged sidecar.

## 14. Sidecar packaging and supply chain

### 14.1 Current tools

- `ffmpeg`: media decoding and PCM output.
- `ffprobe`: stream metadata and duration probing.
- `yt-dlp`: YouTube audio and caption extraction.

`tauri.conf.json` declares each under `bundle.externalBin`. Tauri requires a file
named with the Rust target-triple suffix, for example:

```text
yt-dlp-aarch64-apple-darwin
yt-dlp-x86_64-pc-windows-msvc.exe
yt-dlp-x86_64-unknown-linux-gnu
```

`prepare-sidecars.mjs` selects the current target, copies FFmpeg/FFprobe from
their npm installer packages, downloads a pinned official `yt-dlp` release,
checks SHA-256, makes Unix binaries executable, and runs version probes.

### 14.2 Compatibility matrix

| Target | Current mapping | Confidence | Notes |
|---|---|---:|---|
| macOS ARM64 | `yt-dlp_macos` universal | Tested locally | FFmpeg binary is host-architecture-specific |
| macOS x86_64 | `yt-dlp_macos` universal | CI/native build only | Must test packaged artifact |
| Windows x86_64 | `yt-dlp.exe` | CI/native build only | No real extraction smoke test |
| Windows ARM64 | `yt-dlp_arm64.exe` | Unverified | Not represented in CI |
| Linux x86_64 glibc | `yt-dlp_linux` | CI/native build only | Ubuntu runner only |
| Linux ARM64 glibc | `yt-dlp_linux_aarch64` | Unverified | Not represented in CI |
| Linux musl | Incorrect mapping | Unsupported | Must use musllinux assets |
| Windows x86 | Incorrect mapping | Unsupported | Must use x86 asset or reject target |
| Linux ARMv7 | Incorrect mapping | Unsupported | Must use a matching asset or reject target |

The script currently infers FFmpeg/FFprobe from the build host package. It is not
safe for arbitrary cross-compilation. Release CI should use native builders per
target or an explicit target-to-artifact matrix.

### 14.3 JavaScript challenge runtime

Current `yt-dlp` releases strongly recommend an external JavaScript runtime for
full YouTube support. The official executable contains the EJS component, but
Pawcast does not currently bundle Deno or pass `--js-runtimes`.

Without a runtime, extraction still works for some public videos through limited
clients, but format availability is deprecated and expected to worsen. This is a
release blocker for reliable production support.

Target architecture:

1. Bundle a pinned Deno version for every supported target.
2. Verify its official checksum during preparation.
3. Declare it as a Tauri sidecar.
4. Resolve it through the same native executable locator.
5. pass `--js-runtimes deno:<absolute-sidecar-path>` to both `yt-dlp` calls;
6. run the JavaScript engine in the strongest practical restricted/JIT-less mode;
7. include Deno and EJS licenses in distribution notices;
8. add a CI test that fails if `yt-dlp` reports no supported JS runtime.

## 15. Supported and unsupported video classes

### 15.1 Expected to work

- Public, embeddable, on-demand videos.
- Videos with ordinary audio-only formats exposed to unauthenticated clients.
- Manual captions in the requested language or English.
- Automatically generated captions when YouTube exposes them.
- Videos without captions, provided an AI transcription provider is configured.

### 15.2 Expected to fail or degrade

- Private videos.
- Members-only or paid content.
- Age-gated videos requiring authenticated verification.
- Region-restricted content unavailable from the user's location.
- DRM-protected formats.
- Videos requiring cookies or a PO token.
- Active live streams and some post-live manifests.
- Removed, blocked, or disabled videos.
- Videos that disallow embedding: extraction may work while iframe playback fails.
- Rate-limited guest sessions.
- Videos whose available formats require a JavaScript challenge the packaged
  runtime cannot solve.
- Videos with captions only in a language other than the requested language or
  English; AI fallback will be used.

Authentication is intentionally unsupported. Do not add cookie extraction as a
quick fix: it changes the privacy, account-security, policy, and threat model.

## 16. Error model

### 16.1 Native error codes

| Code | Meaning | Retryability |
|---|---|---|
| `invalid_youtube_id` | ID failed the native allowlist | No |
| `sidecar_missing` | Required executable could not be resolved | No until installation/build fixed |
| `youtube_download_failed` | Downloader returned failure | Depends on video/rate limit/runtime |
| `youtube_audio_missing` | Command succeeded but no acceptable audio file exists | Usually no; inspect tool output |
| `io_error` with `youtube_cache_create` | Cache directory could not be created | Possibly |
| `io_error` with `youtube_cache_read` | Cache directory could not be scanned | Possibly |
| `io_error` with `youtube_cache_approve` | Result could not be canonicalized/approved | Possibly |
| Waveform errors | FFprobe, decode, cancellation, or cache failure | Depends on code |

Native errors must remain sanitized. Detailed stderr belongs in local diagnostic
logs with redaction, not in IPC responses.

### 16.2 User-visible states

Recommended state machine:

```text
idle
  → downloading-audio
  → fetching-captions
  → analyzing-waveform
  → ready

Any active state
  → cancelled
  → recoverable-error (retry offered)
  → terminal-error (reason and supported alternatives shown)
```

Current waveform UI compresses these into idle/analyzing/ready/error. Future UI
should distinguish network download from CPU analysis and show meaningful retry
behavior.

## 17. Performance and resource model

### 17.1 Network

The iframe and extraction paths each request YouTube resources. Loading a video
therefore consumes more network traffic than iframe playback alone. Caption and
audio requests are separate by design.

### 17.2 CPU

`yt-dlp` mostly performs network and manifest work. FFmpeg decoding and waveform
aggregation are CPU-intensive and scale with duration. Analysis runs natively so
it does not block React, but progress events and cancellation remain important.

### 17.3 Memory

Waveform processing streams PCM through the native process. AI fallback currently
loads the complete prepared audio into a renderer `Blob`, which can be expensive
for long videos. Move long-form transcription fully behind the native boundary or
use bounded chunks before production.

### 17.4 Disk

Disk usage includes original extracted audio, caption files, and waveform levels.
Without eviction, growth is unbounded. See the target retention policy.

## 18. Observability

Diagnostics should answer:

- Which stage failed: resolve, download, caption, probe, decode, cache, or AI?
- Which tool version and target triple were used?
- Was a JavaScript runtime available?
- Was the result a cache hit?
- How long did audio download, caption retrieval, and waveform analysis take?
- Was fallback used, and why?

Recommended structured event fields:

```text
operation             youtube_prepare | youtube_caption | waveform_analyze
video_id_hash         irreversible diagnostic hash; never raw URL
target_triple         Rust target
tool_version          yt-dlp/ffmpeg version
cache_hit             boolean
duration_bucket       bounded bucket, not exact viewing history when telemetry is remote
result_code           stable safe code
elapsed_ms            integer
```

Do not send raw video IDs, titles, transcript text, signed URLs, native paths,
cookies, tokens, or downloader output to remote telemetry without an explicit
privacy design and user consent.

## 19. Testing strategy

### 19.1 Unit tests

Required coverage:

- video ID allowlist accepts canonical IDs and rejects traversal/arguments;
- language validation and fallback;
- JSON3 event parsing, missing fields, multiline text, zero duration;
- media ID stability;
- `WaveformLoader` concurrent callers share one native analysis;
- failed preparation is evicted from the frontend promise cache;
- transcript decision tree chooses captions before AI;
- cache scanner rejects partial, caption, symlink, and unexpected files.

### 19.2 Native integration tests

Use fixtures, not YouTube, for deterministic CI:

- a short Opus/WebM audio fixture;
- a short AAC/M4A fixture;
- valid and malformed JSON3 fixtures;
- FFprobe metadata parsing;
- waveform analysis and level readback;
- canonical path approval;
- concurrent native requests against the same video directory.

### 19.3 External smoke tests

A scheduled, non-blocking job may test a small allowlisted public video for:

- metadata extraction;
- audio download;
- FFprobe recognition of an audio stream;
- caption retrieval when a stable captioned fixture is available;
- absence of “no supported JavaScript runtime” warnings;
- end-to-end waveform creation.

External tests are inherently nondeterministic and must not be the only release
gate. Use content owned by the project or explicitly licensed for testing.

### 19.4 Platform release matrix

Each supported release target must run:

1. `npm run prepare:sidecars`;
2. `npm run verify:sidecars`;
3. executable architecture inspection;
4. sidecar version probes;
5. local fixture waveform test;
6. packaged-app launch test;
7. optional external YouTube smoke test.

CI currently covers native macOS, Ubuntu, and Windows hosts but not all advertised
architectures and not real extraction. Unsupported targets should fail during
preparation rather than silently receiving the wrong binary.

Prepare sidecars:

```bash

```

Verify existing sidecars and checksums:

```bash
npm run verify:sidecars
```

Run desktop development:

```bash
npm run dev:tauri
```

Override tool paths for diagnostics or controlled testing:

```text

PAWCAST_FFMPEG=/absolute/path/to/ffmpeg
PAWCAST_FFPROBE=/absolute/path/to/ffprobe
```

Overrides are trusted developer/operator configuration. Never populate them from
user-controlled application settings.

## 21. Sidecar update runbook

When updating `yt-dlp`:

1. Read upstream release notes and YouTube extractor changes.
2. Confirm compatible EJS and JavaScript runtime versions.
3. Enumerate official assets for every supported target.
4. Download the official checksum manifest over HTTPS.
5. Update the pinned release and per-asset SHA-256 values.
6. Re-run sidecar preparation on every native target.
7. Run architecture and version probes.
8. Run deterministic media fixtures.
9. Run the allowlisted public-video smoke test.
10. Test captions, caption failure with audio success, and no-caption AI fallback.
11. Review license/notice changes.
12. Record the update and test evidence in the release notes.

Never switch to an unpinned “latest” URL in a production build. Never accept a
checksum fetched from a different untrusted origin than the release artifact
without signature verification or another independent trust mechanism.

## 22. Production-hardening roadmap

### P0 — required before broad distribution

- Resolve the product/legal decision for third-party content processing.
- Bundle pinned Deno sidecars and pass `--js-runtimes` explicitly.
- Define an explicit supported target matrix and reject everything else.
- Add native per-video locks and atomic completion markers.
- Add process timeouts, cancellation, stderr capture, and safe stage-specific
  errors.
- Implement cache size limits and deletion controls.
- Validate output files canonically inside the expected directory.
- Add packaged-app tests on macOS, Windows, and Linux.

### P1 — reliability and UX

- Expose download/caption/analysis stages separately.
- Add retry with bounded exponential backoff for transient failures.
- Persist transcript provenance and caption language/kind.
- Add caption track discovery and user selection.
- Slice A-B ranges natively before AI transcription.
- Avoid loading full long-form audio into renderer memory.
- Revalidate stale captions and media metadata.
- Add cache hit/miss and stage timing diagnostics.

### P2 — quality improvements

- Detect audio/iframe duration drift before displaying waveform alignment.
- Support alternate audio-track selection where technically and legally allowed.
- Merge fragmented caption events into sentence-oriented segments without losing
  timing.
- Preserve manual versus automatic caption provenance.
- Add resumable/cancellable downloads where supported.
- Add background cache maintenance and disk-pressure handling.

## 23. Rejected alternatives

### 23.1 Read audio from the iframe

Rejected because the iframe API intentionally exposes playback controls, not
cross-origin media bytes or Web Audio nodes.

### 23.2 Fetch Googlevideo URLs from the browser

Rejected because signed URLs are unstable, cross-origin behavior is unsuitable,
and exposing extraction logic in the renderer weakens the security boundary.

### 23.3 Use the official captions API for arbitrary public videos

Rejected because caption download requires OAuth and permission to edit the
video. It remains viable only for a future owner-authorized workflow.

### 23.4 Show a synthetic waveform

Rejected as the primary desktop behavior because it misrepresents audio and
cannot support accurate phrase selection. A clearly labeled placeholder is only
acceptable when the native capability is unavailable.

### 23.5 Play the downloaded audio instead of the iframe

Rejected because it would replace the official playback surface, introduce
policy and synchronization concerns, and require Pawcast to own adaptive media
playback behavior.

### 23.6 Automatically use browser cookies

Rejected because it expands access to restricted content, exposes account
credentials to a third-party executable, creates account-ban risk, and materially
changes the privacy and threat model.

## 24. Architectural invariants checklist

Before merging changes to this feature, verify:

- [ ] Playback still uses the YouTube iframe.
- [ ] Raw user URLs never cross into the native downloader.
- [ ] Native ID and language validation occurs before filesystem/process work.
- [ ] No shell command construction was introduced.
- [ ] Audio success is independent from caption success.
- [ ] Concurrent preparation and waveform calls deduplicate safely.
- [ ] Cached paths remain under the active data directory and require approval.
- [ ] Caption text is rendered as untrusted text, not HTML.
- [ ] Errors are sanitized and do not expose native paths or signed URLs.
- [ ] Sidecars are target-correct, pinned, checksum-verified, and executable.
- [ ] Supported targets have packaged-app validation.
- [ ] Cache growth and stale artifacts are considered.
- [ ] Transcripts record or preserve source provenance where available.
- [ ] The legal/product authorization boundary has not silently expanded.

## 26. Decision record

The accepted design is:

> Keep official iframe playback as the user-visible media source. On desktop,
> prepare a separate, validated, locally cached audio analysis artifact through
> pinned native sidecars. Prefer source captions, fall back to AI transcription,
> and reuse the native waveform pipeline. Treat extraction as a constrained,
> failure-prone capability rather than an extension of the iframe API.

This decision should be revisited if YouTube provides an official API that grants
authorized access to audio bytes or public caption tracks for this use case.
